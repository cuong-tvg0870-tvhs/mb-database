## What this repo is

`mb-db` (npm name `mb-migrations`) is the **single source of truth for the Prisma schema and migrations** of the "MB" Meta/Facebook Ads management platform. It has almost no runtime code — its job is to own `prisma/schema.prisma`, generate the Prisma client, run migrations against Postgres, and **propagate the schema to sibling repos** that actually serve traffic.

The platform is a set of sibling repos under `../` (i.e. `/Users/cuongdangquoc/Documents/thanhvinh-source.nosync/new/`):

- **`mb-ads`** (npm name `ad-management`) — NestJS. The main API; publishes drafts to Meta, serves the dashboard/detail screens, runs the AI assistant.
- **`mb-batch`** — NestJS. Background jobs: meta-sync (pulls Campaign/AdSet/Ad/insight data from Meta), media upload, automation runner.
- **`mb-frontend`** (npm name `my-v0-project`) — Next.js UI. Talks to `mb-ads` over HTTP; has **no** Prisma schema of its own.

`mb-ads` and `mb-batch` both connect to the **same Postgres database** and both hold a *copy* of this schema at `<repo>/prisma/schema.prisma`. Those copies are generated by `sync-schemas.js` in this repo — never hand-edit them.

## The critical workflow: schema changes are made HERE, then synced

Any schema change starts in `prisma/schema.prisma` in this repo. The golden path:

```bash
# 1. Edit prisma/schema.prisma here
# 2. Create the migration with a DESCRIPTIVE name, then push the schema to siblings:
npx prisma migrate dev --name add_something_descriptive
node sync-schemas.js
```

- `npm run migration:run` does `prisma migrate dev --name init && node sync-schemas.js`. The `--name init` is hardcoded, which is why the *older* migration folders are all named `*_init`. Recent practice is to run `prisma migrate dev --name <descriptive>` **manually** (see the descriptive names on recent migrations), then `node sync-schemas.js` — prefer this over `npm run migration:run`.
- `sync-schemas.js` copies `prisma/schema.prisma` into `../mb-ads/prisma/schema.prisma` and `../mb-batch/prisma/schema.prisma` (stripping the `output = ...` line from the `generator client` block), then runs `npx prisma generate` in each if their `node_modules` exists. If those repos aren't checked out as siblings, sync is skipped with a warning.

**Parity is the central operational constraint.** Because two independent writers (`mb-ads` and `mb-batch`) share one DB and one client shape, the schema in all three repos must stay identical, and application-level field lists that mirror the schema (e.g. meta-sync `rawPayload` field lists, insight column maps) must be updated in *both* backends. Migrations should be **additive and non-breaking** wherever possible — dropping/renaming a column a running writer still uses will break production. Migration `.sql` files in this repo are routinely annotated (in Vietnamese) with the intent, the back-compat reasoning, and rollout notes; follow that convention.

## Commands

```bash
npm run migration:generate   # npx prisma generate (regenerate local client only)
npm run migration:run        # prisma migrate dev --name init && node sync-schemas.js
npm run seed                 # ts-node prisma/seed.ts (admin/employee users + one dropdown)
npm run seed-dropdown        # ts-node prisma/dropdown-seed.ts (dropdown categories/items)
node sync-schemas.js         # push current schema to mb-ads + mb-batch and regenerate their clients
```

There are no tests or linters in this repo — it's schema + migrations + seeds. Validation is `npx prisma validate` / `npx prisma generate`, and downstream `npx tsc` in `mb-ads` / `mb-batch` after a sync to confirm the client shape still typechecks.

## Prisma 7 specifics (easy to trip on)

- **Prisma 7.0.1 with driver adapters.** The datasource block in `schema.prisma` declares only `provider = "postgresql"` — there is **no `url`** there. The connection string is supplied by `prisma.config.ts` via `env('DATABASE_URL')`, and at runtime `src/prisma/prisma.service.ts` builds a `pg.Pool` + `@prisma/adapter-pg` (`PrismaPg`) and passes it as `{ adapter }` to `PrismaClient`. Don't "fix" the missing url in the schema.
- **Generated client output:** `src/generated/prisma` (gitignored). Seeds import types/client from `../src/generated/prisma`; `sync-schemas.js` removes the `output` line so each sibling generates into its own default location.
- **Raw-SQL migrations exist.** Some tables were added with hand-written SQL rather than `prisma migrate dev` (the schema comments flag this, e.g. the automation insight cache: "Migration `add_automation_insight_cache` (raw SQL — KHÔNG dùng `prisma migrate dev`)"). Don't assume every model has a matching diff-generated migration.
- Config files: `prisma.config.ts` (schema + migrations paths + datasource), `tsconfig.json` targets ES2023 / nodenext.

## Schema domain map

`prisma/schema.prisma` is ~83 models / ~34 enums. The main clusters (in file order):

- **Identity & permissions:** `User`, `UserActivity`, `Role`, `Permission`/`RolePermission`, plus per-resource ACLs (`UserFanpage`, `UserProductCatalog`, `AccountMember`, `FolderMember`).
- **Meta ad hierarchy (live/synced):** `Account` → `Campaign` → `AdSet` → `Ad` → `Creative` (+ `AdImage`, `AdVideo`, `CidGroup`). Meta-sync in `mb-batch` writes these.
- **Insights (parallel per level):** `CampaignInsight`, `AdSetInsight`, `AdInsight`, `CreativeInsight`, and audience-breakdown variants (`CampaignAudienceInsight`, `AdsetAudienceInsight`, `AdAudienceInsight`). Heavy tables; watch query cost.
- **Draft / publish layer ("System*"):** `SystemCampaign`, `SystemAdSet`, `SystemAd`, `SystemCreative`, `TemplateCampaign`, `PublishHistory`, `DraftAutomationHistory` — the draft-builder state that `mb-ads` publishes to Meta. Much config rides JSON columns (e.g. `SystemAdSet.data`) rather than dedicated columns.
- **Media pipeline:** `CreativeFolder`, `CreativeAsset`, `CreativeAssetMapping`, `LarkRecord` (CID ↔ product mapping from Lark), `DriveFile`. CID = the content identifier that links filenames to products.
- **Automation ("Care Ads", Birch-style rule engine):** the large `Automation*` block (`AutomationRule`, `AutomationFilterGroup`/`AutomationFilter`, `AutomationTask*`, `AutomationSchedule`, `AutomationRuleRun`/`AutomationRuleRunItem`, `AutomationInsight*` cache, `CustomMetric`, `MetaConnection`). Deliberately kept separate from the legacy models above ("Không chạm vào models legacy").
- **Projects & org:** `Project`, `ProjectMember`, `ProjectAccount`/`ProjectFanpage`/`ProjectProductCatalog`/`ProjectFolder`, `ProductCatalog`/`ProductSet`/`ProductFeed`.
- **Platform/UX:** `Contribution` (user feedback/góp ý), `Help*` + `Ai*` (help docs + AI assistant sessions/actions), `SavedFilter`, `ExportPreset`, `DropdownCategory`/`DropdownItem`, `SystemConfig`, `EntitySeen` (per-user "đã xem" badges), `ChangeLog`, `AccountProposal`.

## Inspecting production data

The production Postgres is reachable via an SSH tunnel at `127.0.0.1:5433`, database `database_mb_production_1`. `psql` against that tunnel is the usual way to check real data when diagnosing a schema/data question (credentials live in `.env`, not in this file). When reasoning about how a column is actually populated, remember it may be written by `mb-ads`, `mb-batch`, or both — grep both backends.
